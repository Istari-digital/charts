{{- if .Values.alpha.acl.enabled }}
   ACL is ENABLED, so every client must log in.
   {{- if .Values.alpha.acl.bootstrap.enabled }}
   The bootstrap Job rotated `groot` off its default password and provisioned the
   declared accounts. Log in with a named account (for example `istari-admin`),
   using its password from your credentials store:

       kubectl -n {{ include "dgraph-sec.namespace" . }} port-forward svc/{{ include "dgraph-sec.alpha.fullname" . }} 8080:8080 &
       curl -s localhost:8080/admin -H 'Content-Type: application/json' \
         --data '{"query":"mutation($u:String!,$p:String!){login(userId:$u,password:$p){response{accessJWT}}}","variables":{"u":"istari-admin","p":"<password>"}}'

   To add an account: append to `alpha.acl.bootstrap.users` (a service also needs a
   group with predicate rules under `alpha.acl.bootstrap.groups`), add its password
   to the credentials Secret, then re-apply. The reconciler converges it next deploy.
   {{- else }}
   Set `alpha.acl.bootstrap.enabled=true` with a credentials Secret to auto-rotate
   `groot` and provision accounts; otherwise rotate `groot` manually before use.
   {{- end }}
{{- else }}
   This release deploys a WORKING cluster out of the box: with ACL disabled no
   login is required, so applications can connect immediately. To make it
   production-grade, layer on the security features above:

     1. Authentication — enable ACL and auto-provision accounts:
          --set alpha.acl.enabled=true \
          --set alpha.acl.existingSecret=<secret-with-hmac-and-passwords> \
          --set alpha.acl.bootstrap.enabled=true
     2. Encryption / TLS — see values.yaml `alpha.encryption` and `alpha.tls`
        (use scripts/make_tls_secrets.sh to generate TLS material).
     3. Backups — enable a schedule and point it at S3/MinIO/NFS:
          --set backups.full.enabled=true \
          --set backups.destination=s3://s3.<region>.amazonaws.com/<bucket>
        If ACL is on, you MUST also set backups.admin.user and a password
        (see the SECRET HANDLING note in values.yaml — prefer a managed Secret).
     4. NetworkPolicy — restrict access on a CNI that enforces it:
          --set networkPolicy.enabled=true
{{- end }}

2. Get the Dgraph Alpha HTTP/S endpoint by running these commands.

{{- if .Values.global.ingress.enabled }}

    echo "Access Alpha HTTP/S service: http://{{ .Values.global.ingress.alpha_hostname }}"

{{- else if .Values.alpha.ingress.enabled }}

    echo "Access Alpha HTTP/S service: http://{{ .Values.alpha.ingress.hostname }}"

{{- else if contains "NodePort" .Values.alpha.service.type }}

    export ALPHA_NODE_PORT=$(kubectl get --namespace {{ include "dgraph-sec.namespace" . }} -o jsonpath="{.spec.ports[?(@.name=='http-alpha')].nodePort}" services {{ include "dgraph-sec.alpha.fullname" . }})
    export ALPHA_NODE_IP=$(kubectl get nodes --namespace {{ include "dgraph-sec.namespace" . }} -o jsonpath="{.items[0].status.addresses[0].address}")
    echo "Access Alpha HTTP/S service using http://$ALPHA_NODE_IP:$ALPHA_NODE_PORT"
{{- else if contains "LoadBalancer" .Values.alpha.service.type }}
   NOTE: It may take a few minutes for the LoadBalancer IP to be available.
         You can watch the status of by running 'kubectl get --namespace {{ include "dgraph-sec.namespace" . }} service --watch {{ include "dgraph-sec.alpha.fullname" . }}'

    export ALPHA_SERVICE_IP=$(kubectl get service --namespace {{ include "dgraph-sec.namespace" . }} {{ include "dgraph-sec.alpha.fullname" . }} --template "{{"{{ range (index .status.loadBalancer.ingress 0) }}{{.}}{{ end }}"}}")
    echo "Access Alpha HTTP/S service using http://$ALPHA_SERVICE_IP:8080"

{{- else if contains "ClusterIP" .Values.alpha.service.type }}

    export ALPHA_POD_NAME=$(kubectl get pods --namespace {{ include "dgraph-sec.namespace" . }} --selector "statefulset.kubernetes.io/pod-name={{ include "dgraph-sec.alpha.fullname" . }}-0,release={{ .Release.Name }}" --output jsonpath="{.items[0].metadata.name}")
    echo "Access Alpha HTTP/S using http://localhost:8080"
    kubectl --namespace {{ include "dgraph-sec.namespace" . }} port-forward $ALPHA_POD_NAME 8080:8080

{{- end }}

   NOTE: Change "http://" to "https://" if TLS was added to the Ingress, Load Balancer, or Dgraph Alpha service.

CONNECTING ANOTHER SERVICE IN THE CLUSTER

   This configuration requires setup before an in-cluster client can reach Alpha at
   {{ include "dgraph-sec.alpha.fullname" . }}.{{ include "dgraph-sec.namespace" . }}.svc{{ include "dgraph-sec.domainSuffix" . }} on 8080 (HTTP) / 9080 (gRPC):
{{- if .Values.networkPolicy.enabled }}

   * NetworkPolicy is ENABLED — only pods carrying every label in
     networkPolicy.clientPodLabels may reach 8080/9080.
{{- if .Values.networkPolicy.clientPodLabels }}
     Label client pods with:
{{- range $k, $v := .Values.networkPolicy.clientPodLabels }}
         {{ $k }}: {{ $v }}
{{- end }}
{{- else }}
     networkPolicy.clientPodLabels is EMPTY — no client pods are admitted yet. Set it
     (or add a rule to networkPolicy.extraIngress) before any client can connect.
{{- end }}
{{- end }}
{{- if .Values.serviceMesh.enabled }}

   * Service mesh is ASSUMED — the client pod must be in the mesh (carry a sidecar),
     and the mesh AuthorizationPolicy must admit its service-account principal. The
     chart ships no Istio CRDs; that policy lives in the platform mesh.
{{- end }}
{{- if and (not .Values.serviceMesh.enabled) .Values.alpha.tls.enabled }}

   * Native TLS is ON — the client must connect over TLS and trust the chart CA
     (ca.crt from the alpha TLS Secret){{ if .Values.alpha.tls.clientAuthType }}, and present a client certificate (clientAuthType={{ .Values.alpha.tls.clientAuthType }}){{ end }}.
{{- end }}
{{- if .Values.alpha.acl.enabled }}

   * ACL is ON — the client must also log in (see above) and send the returned access
     token on every request.
{{- end }}

   Step-by-step for each: docs/topology.md, "Letting another in-cluster service connect".
